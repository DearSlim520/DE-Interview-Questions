## Q1. Kappa vs Lambda

> 📌 **频率**: 2025 · ★★☆

### 🎯 架构对比

```
┌─── Lambda Architecture ───────────────────────────────┐
│                                                       │
│   Source ──► Batch Layer (Hive/Spark) ──┐             │
│         │                               ├──► Serving  │
│         └► Speed Layer (Flink/Storm) ──┘             │
│                                                       │
│   ⚠️ 两套代码 · 两套口径 · 结果需合并               │
└───────────────────────────────────────────────────────┘

┌─── Kappa Architecture ────────────────────────────────┐
│                                                       │
│   Source ──► Stream Layer (Flink) ──► Serving         │
│                    │                                  │
│                    └── 回放 Kafka/Paimon 做离线复算    │
│                                                       │
│   ✅ 单一代码 · 单一口径 · 统一可信源                │
└───────────────────────────────────────────────────────┘
```

### 📊 详细对比

| 维度 | Lambda | Kappa |
|------|--------|-------|
| 代码维护 | 两套（批+流） | 一套 |
| 数据一致性 | 需对账合并 | 天然一致 |
| 离线兜底 | 有（批层） | 回放 Kafka 重算 |
| 复杂度 | 高 | 低 |
| 落地依赖 | 传统大数据栈 | Flink + 湖仓存储 |

### 💡 类比记忆

> - **Lambda** = 同时养"白班团队"和"夜班团队"做同一份报表，结果还要对账 🤦
> - **Kappa** = 只一个团队 24h 流水线，需要历史数据时从备份磁带（Kafka/Paimon）从头跑一遍 🏃

### 📝 痛点案例

> 某电商订单口径，离线和实时差 0.3%，业务方天天追着对账
> → 切 **Paimon + Flink Kappa** 后，单一可信源，口径对齐

### ✅ 2025 趋势：为什么大家抛弃 Lambda？

- **Flink** 成熟 → 流处理能力足够
- **Iceberg / Paimon** → 流写 + 批读 + 时间旅行
- **湖仓一体** → 一份存储同时服务实时 & 离线

### 🧠 记忆锚点

```
两条链路 → 一条链路 + 可回放
```
